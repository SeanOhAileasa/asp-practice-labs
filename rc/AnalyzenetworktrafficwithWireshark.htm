Exercise 1 - Download and Install Wireshark

Wireshark is a free open-source packet sniffer useful for analyzing network traffic. Similar to TCPdump, it uses a powerful and a user-friendly GUI that greatly improves and simplifies its usage for network traffic analysis. Like TCPdump, Wireshark uses the libpcap or WinPcap library to capture and store traffic information. Learning to use WireShark is easy as the the Interface remains consistent whether you use it on Windows, Linux or Mac.

A key difference to log files, however, is that Pcap files store the contents of packets which are captured. Logs can be misleading as they can tell you what events happened in terms of processes but not what the client specifically requested from the host.

Please refer to your course material or use your favorite search engine to research for more information about this topic.

Task 3 - Use Wireshark Launch Modes

There are two different launch modes used in Wireshark:

  • The first mode is launched by the default user who has restricted privileges.
  • The second mode requires administrative privileges. Administrative privileges permit a greater level of capture by allowing Wireshark to view all traffic on the Ethernet adapter.

Further, there is legacy mode and a newer interface for Wireshark. You will briefly explore those options and differences.

Task 1 - Analyzing captured information

In this task, you will explore the details of traffic flows and how to understand them using the main interface. Also, you will list the information according to different tabs and then move it into basic filtering techniques. Knowledge of protocols is essential to understanding all the information in Wireshark.

Alert: At this point, it’s a good idea to read up on protocols using your favorite search engine or refer to the educational text to get a better understanding of them.

Step 1

Now let’s explore some of the interface and options you have in Wireshark.

First, notice the display options:

  • No - Packet Number which is counted for each packet recorded
  • Time - Time of the packet recording
  • Source - Source IP address
  • Destination - Destination IP address
  • Protocol - Which type of protocol has been used
  • Length - Length of the Packet
  • Info - Any extra information which could be useful

First, click on the Tab Protocol in order to list the information by type. That will list the ARP protocols first, and you will quickly be able to discover devices making requests for information by checking the Info tab.

You need to scroll through the details to find the results of the ARP sessions. From this method, all the communicating devices that are calling out to check each other’s IP addresses, you learn that all the addresses listed for all switched on devices which are connected to the same Ethernet adapter. You can see calls for 192.168.02, which is the PLABDM01 server asking for information about who has 192.168.0.1, which is the PLABDC01 server.

You also see devices establishing their addresses. For example, the device 192.168.0.4, which is the PLABWIN10 server, confirms its address location at 00:15:5d:04:55:69.

Step 2

Scroll a little further down and search for the device you rebooted.

Its frame can be seen making a Gratuitous Request via a Broadcast effectively setting its source and destination address information against its IP.

If another machine did own 192.168.0.5, PLABWIN801 would detect a duplicate IP address and prompt the user to change the configuration.

Wireshark decodes the OUI and that you can expand the MAC fields to decode the multicast/broadcast bit and a locally administered bit.

The ARP headers (layer 2.5 or 3-ish) contain similar information, plus the sender and target IP addresses. ARP is a very simple protocol. IP and higher-level packets often contain many more headers.

Also, note the bottom frame. This contains the raw data in a hexadecimal format (the computers receive it as a series of 1s and 0s. When you select information in pane 2, the relevant hex digits are selected here (and vice versa).

Step 3

Let’s explore the capture a little further. Scroll throughout the capture and view the contents of the protocols collected overall.

Notice the color coding which Wireshark uses to clarify the different traffic types.

Color Coding - You’ll probably see packets highlighted in green, blue, and black. Wireshark uses colors to help you identify the types of traffic at a glance.

By default, the color coding is as follows -

  • Green - TCP traffic
  • Dark blue - DNS traffic
  • Light blue - UDP traffic
  • Black - TCP packets with problems — for example, they could have been delivered out-of-order

Type the following in the filter bar and press Enter:

dnsserver

You will see queries going out to PRACTICELABS.COM which is the domain, and another query is being made for updates, for example, to Microsoft.

Note: If you are unable to get any result when searching for dnsserver, you can try to search for the keyword - dns. This task uses the keyword - dnsserver.

Step 4

Clear the Filter command by deleting the filter syntax and press the Arrow to come back to the original capture.

Step 5

Moving down the list, you can view plenty of TCP traffic from all the devices that are using that protocol.

There is a lot of information on TCP that is flowing, which can be overwhelming to inspect.

Step 6

Let’s break some of it down and see what it contains.

Now, type the following into the filter bar and press Enter:

http

This will display HTTP traffic.

You can then investigate any IP devices which have been using that well-known port and look at internet traffic coming and leaving the devices.

Again, right-click one of the streams, select Follow and then select TCP stream.

Step 7

The output is displayed. Click Close.

Step 8

Type the following in the filter bar and press Enter:

You can view all the traffic which has flowed from PLABDC01 since the recording began. Remember to click on the No. tab to order the results as they were captured.

Exercise 6 - Perform Packet Jumping

Packet Jumping is a process which involves the coordination of different frames of reference within a capture. Wireshark logs which packets arrive in which order. It then places markers on these packets so that a quick link method can be used to snap to the frame for ease of use.

Task 1 - Perform Packet Jumping

In this task, you will learn how to jump packets which link together different frames. When you capture traffic, you will find there is a captured packet that relates to another captured packet. When you open the first captured packet, it would provide the reference to the second captured packet. Without leaving the description of the first captured packet, you can navigate to the second captured packet or vice versa.

To perform packet jumping, perform the following steps:

Step 1

Connect to PLABSA01.

On PLABSA01, click the PCAPS folder on the desktop and open the Lab folder.

Double-click the Security Lab 2 capture.

Type the following in the Filter field and press Enter:

icmp

Step 2

Scroll down to the pink area in the scrollbar.

Here, you can see some ping (ICMP) requests which have been made.

Step 3

There can be many details in traces; ping often details the request and replies with the data.

Click on Frame 914 and expand the Packet Details Pane.

Expand Internet Control Message Protocol. Also, expand Data below Internet Message Protocol.

Notice that this request is linked directly to frame 915 as it holds the response.

Step 4

Now, double-click on Frame 915 to view the same information. This frame is highlighted in a blue color.

Here you see the response to 914 frames took 0.106ms to reply.

Step 5

Now, double-click on the blue value of:

Request Frame:914

The reader will jump to 914. The output in the bottom pane displays the reference of Response frame: 915.

View this carefully to see Wireshark move the packets which are linked together.

Step 6

Repeat again and double click on:

Response Frame:915

Wireshark jumps to that packet and shows the reference of Request frame: 914.

Clear the filter and click Enter to return to the normal screen.
